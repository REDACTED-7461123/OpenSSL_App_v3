**Предмет исследования:** Разработка практически ориентированного приложения для криптографической защиты файлов, предназначенного для пользователей без специальной подготовки в области информационной безопасности.

**Цель работы:** Разработать доступное настольное приложение, реализующее криптографические операции (шифрование/расшифрование файлов, вычисление хешей, генерацию RSA-ключей) с интуитивным графическим интерфейсом и ведением журнала операций.

**Основные результаты:** 
- Реализовано приложение на платформе C# / WPF / .NET 10 с поддержкой пяти основных криптографических операций;
- Разработан механизм валидации пароля с оценкой стойкости;

- Проведено комплексное тестирование функциональности приложения.

**Область применения:** Образовательные учреждения, корпоративный сектор, конечные пользователи для защиты персональных данных и контроля целостности файлов.

**Практическая значимость:** Приложение снижает барьер входа при использовании криптографических методов защиты, минимизирует ошибки пользователей через валидацию и подсказки, обеспечивает прозрачность криптографических операций.

# 1. Используемые термины и сокращения

**ИБ (информационная безопасность)** — состояние защищённости информации и инфраструктуры от угроз нарушения конфиденциальности, целостности и доступности.

**Конфиденциальность** — свойство информации быть доступной только тем, кто имеет право доступа.

**Целостность** — свойство информации сохранять корректность и неизменность при хранении и передаче (или, если изменения происходят — быть обнаружимыми).

**Аутентичность (подлинность)** — подтверждение, что объект (например, файл) соответствует ожидаемому источнику и не был подменён.

**Шифрование** — криптографическое преобразование данных в форму, непонятную без ключа, с целью защиты конфиденциальности.

**Расшифрование** — обратное преобразование, возвращающее исходные данные при наличии ключа.

**Симметричное шифрование** — шифрование, в котором для шифрования и расшифрования используется один и тот же секретный ключ.

**Асимметричное шифрование (RSA)** — шифрование на паре ключей (открытый/закрытый), где открытый ключ используется для шифрования или проверки, а закрытый — для расшифрования или подписи.

**Хеш-функция** — функция, отображающая входные данные произвольной длины в выход фиксированной длины (хеш). Обычно используется для проверки целостности.

**Хеш-сумма** — результат вычисления хеша для файла; при изменении файла хеш обычно меняется, что позволяет обнаружить подмену.

**CSV** — текстовый формат для экспорта таблиц (Comma-Separated Values), удобен для просмотра истории операций в Excel/табличных редакторах.

**UX/UI** — удобство использования и интерфейс пользователя.

**HTTPS** - (HyperText Transfer Protocol Secure) - зашифрованный протокол передачи гипертекста

**CLI** - (Command Line) командная строка

**Троян** - Вирус, который прячется внутри безобидного на первый взгляд архива/файла

**C# (CSharp)** — объектно-ориентированный язык программирования, разработанный компанией Microsoft для платформы .NET.

**WPF (Windows Presentation Foundation)** — фреймворк для создания графических приложений на платформе Windows с использованием декларативного языка XAML.

**.NET** — кроссплатформенная платформа разработки приложений от Microsoft, обеспечивающая выполнение управляемого кода.

**XAML (eXtensible Application Markup Language)** — язык разметки, используемый в WPF для описания интерфейса пользователя.

**GUI (Graphical User Interface)** — графический интерфейс пользователя, взаимодействие между человеком и приложением через визуальные элементы.

**OpenSSL** — открытая криптографическая библиотека, реализующая криптографические алгоритмы и протоколы безопасности.

**Process** — класс в C# для управления процессом и его взаимодействия с внешними приложениями.

**ProcessStartInfo** — класс в C# для конфигурирования параметров запуска внешнего процесса.

**JSON (JavaScript Object Notation)** — текстовый формат обмена данными, легко читаемый человеком.

**JSONL (JSON Lines)** — вариант JSON-формата, где каждая строка представляет отдельный JSON-объект, удобен для логирования.

**Асинхронность** — в программировании: способность приложения выполнять операции без блокировки основного потока.

**Валидация** — проверка корректности и соответствия входных данных установленным требованиям.

**Верификация** — процесс проверки соответствия результатов ожидаемым результатам; в криптографии — подтверждение подлинности.

**Архитектура приложения** — структурная организация компонентов приложения и способы их взаимодействия.

# 2. Введение

Современный пользователь ежедневно обрабатывает файлы: документы, фотографии, архивы, резервные копии, учебные и рабочие материалы. Эти файлы часто пересылаются через мессенджеры, почту, облачные хранилища, переносные носители. При этом возникают типовые риски:

- утечка конфиденциальной информации при отправке «важного файла»;
- потеря контроля над тем, кто может открыть файл после пересылки;
- подмена скачанного файла (например, установщика) и заражение компьютера;
- невозможность быстро и корректно проверить целостность данных без специальных знаний.

На практике сильные криптографические инструменты существуют давно, но проблема заключается не в отсутствии криптографии, а в **пороге входа**: большинство решений либо требуют уверенного владения командной строкой и серьёзных знаний в ИБ, либо перегружены профессиональными настройками, либо не дают понятных подсказок и сценариев использования.

**Цель проекта:** разработать доступное неспециалисту приложение, позволяющее выполнять типовые криптографические операции с файлами (шифрование/расшифрование, генерация RSA-ключей, хеширование и сравнение хешей) и тем самым повышающее уровень практической защищённости пользователя.

**Задачи проекта:**
1. Исследовать пользовательскую проблему и типовые угрозы при хранении и передаче файлов.
2. Проанализировать существующие решения (аналоги и прототипы), выявить их недостатки для целевой аудитории.
3. Сформулировать требования к продукту: функциональные, нефункциональные, UX-требования.
4. Спроектировать архитектуру приложения на C# / WPF / .NET 10 и реализовать прототип/продукт.
5. Реализовать журнал операций (история), экспорт в CSV и смену темы (тёмная/светлая).
6. Провести тестирование и верификацию корректности (в т.ч. на сценариях «подмена файла» и «проверка хеша»).
7. Оценить потенциал внедрения и перспективы развития.

# 3. Исследование проблемы

## 3.1. Предметная область и общий контекст ИБ

Криптография является одним из фундаментальных направлений ИБ. На уровне пользователя она применяется в двух основных «повседневных» задачах:

1) **Защита конфиденциальности файлов**  
Если файл попадёт к третьим лицам (по ошибке отправки, утечке из облака, доступу к флешке, компрометации аккаунта), шифрование должно не позволить прочитать содержимое без ключа.

2) **Проверка целостности и подлинности файлов**  
При скачивании из интернета или получении от другого человека важно понять, не был ли файл изменён. Проверка хеш-суммы — простой и быстрый метод: если хеш совпадает с эталоном, вероятность подмены резко уменьшается.

Однако для неспециалиста эти принципы часто остаются «теорией», потому что:
- сложно выбрать алгоритм и формат ключа;
- непонятно, как безопасно хранить ключ/пароль;
- непонятно, как правильно сравнивать хеш и что делать при несовпадении;
- отсутствует понятная история действий (лог), чтобы вспомнить «чем и когда шифровал».

Следовательно, предметная область проекта: **прикладная криптография для массового пользователя**, включая UX-поддержку и снижение человеческих ошибок.

---

## 3.2. Описание проблемы (где и как возникает)

### Сценарий A: «Нужно отправить важный файл»
Пользователь хочет отправить документ (например, скан паспорта, договор, ведомость, личные фото). Он отправляет файл через почту/мессенджер. Риски:
- пересланный файл может быть перехвачен или открыт на чужом устройстве;
- файл может попасть в общий доступ (ошибочный адресат, общий чат, утечка из облака);
- пользователь не понимает, как правильно «запаролить» файл и какой формат надёжный.

### Сценарий B: «Скачал файл и не уверен в его подлинности»
Пользователь скачивает установщик, архив с материалами или документ. Риски:
- подмена файла на зеркале или заражённой странице;
- загрузка «похожего» файла со встроенным вредоносным кодом;
- пользователь не умеет проверять хеш-суммы, а если и умеет — нет удобного инструмента.

### Итог проблемы
Проблема состоит не только в «отсутствии криптографии», а в **отсутствии доступного средства**, которое:
- снижает вероятность ошибок человека,
- поддерживает стандартные понятные сценарии,
- даёт историю операций,
- оформлено как готовый продукт, а не набор команд.

## 3.3. Актуальность и обоснование проблемы

Актуальность определяется следующими факторами:
- рост объёмов передачи файлов через интернет и облака;
- распространение вредоносных подмен (троянизированные архивы/инсталляторы);
- повышение ценности персональных данных и конфиденциальных документов;
- низкая криптографическая грамотность массового пользователя.

Анализ уязвимостей, проведённый в рамках исследовательской практики, подтверждает масштаб и критическую важность исследования защищённости. Так, по данным Positive Technologies, 84 % организаций экспонируют высоко-рискованные уязвимости на внешних сетевых периметрах, а около одной трети организаций фиксируют попытки эксплуатации известных уязвимостей в своей инфраструктуре. Кроме того, эксперты выявили, что примерно 30 % уязвимостей нулевого дня имели критический уровень опасности при тестировании. Публичная база компании содержит более 300 000 уязвимостей, что указывает на масштаб проблемы и необходимость практической защиты.

## 3.4. Сбор информации по проблеме: распространённость, запрос пользователей, аудитория

### Распространённость
Практически в любой учебной/рабочей среде есть обмен файлами и необходимость:
- ограничить доступ к содержимому;
- убедиться в целостности материалов;
- убедиться что файл не был изменён.

### Наличие запроса пользователей
Запрос проявляется типовыми вопросами:
- «как запаролить файл надёжно?»  
- «как проверить, что скачанный файл не подменили?»  
- «как сравнить хеши двух файлов?»  
- «как сделать это без командной строки?»

### Описание аудитории
Целевая аудитория — **не специалисты ИБ**:
- школьники и студенты (часто скачивают непроверенные программы);
- офисные пользователи (пересылка важных документов и отчётов);
- домашние пользователи (резервные копии, личные файлы);

## 3.5. Предпроектное исследование: анализ прототипов и аналогов 

Перед разработкой важно оценить, какие решения уже существуют, и почему они не закрывают задачу полностью для целевой аудитории.

### 1: инструменты командной строки (CLI)
**Плюсы:** мощные, гибкие, стандарт де-факто для специалистов.  
**Минусы для аудитории:** сложно, высокий риск ошибки (параметры, режимы, кодировки), отсутствует визуальная история.

### 2: Kleopatra
**Описание:** Это мощный инструмент, но он перегружен сложными функциями, требует много времени для того чтобы разобраться. Kleopatra изначально разработана для решения другой задачи - хранение и управление приватными ключами.

**Плюсы:** Много функций, полезно для продвинутых пользователей.  
**Минусы:** Интерфейс ориентирован на опытных пользователей, избыточен для бытовых сценариев («зашифровать файл → отправить»), требует понимания модели PGP и управления ключами, отсутствует упрощённый сценарий быстрой проверки целостности скачанного файла по эталонному хешу.

### 3: Заброшенные проекты с GitHub'а

**Минусы:** Нету поддержки, ни одно решение не доведено до конца.

### 4: Веб-приложение CrypTool
**Плюсы:** Полезно для обучения OpenSSL CLI.  
**Минусы:** Направлено на изучение командной строки, а не на постоянное использование. Это онлайн-приложение, а не готовое к использованию приложение. Отсутствует история.

## 3.6. Предложения решения: авторская концепция и выбор оптимальной идеи

### Выявленная потребность
Пользователю нужен простой инструмент:
- выбрать файл → выбрать действие → получить понятный результат,
- без необходимости изучать командную строку и криптографические термины,
- с минимизацией ошибок и с «памятью» (историей).

### Авторская концепция проекта
**Ключевая идея:** сделать криптографию «как бытовую функцию» — так же просто, как «распечатать файл» или «сжать архив», но при этом корректно и безопасно.

Концепция реализуется через:
- Приложение написано на языке C# с использованием фреймворков WPF и .NET 10;
- Для криптографии использовалась библиотека OpenSSL;
- Внутри приложение вызывает Process() через ProcessStartInfo(), обращается к openssl.exe и выполняет команду в зависимости от выбранной функции.

### Почему выбран именно такой подход
Написание приложения на XAML занимает меньше всего времени, при этом очень удобно контролировать весь интерфейс.
Библиотека OpenSSL выбрана из-за высокой популярности и хорошей подробной документации.

# 4. Решение проблемы

## 4.1. Формализация требований и модели угроз

### Формулировка темы
**Тема проекта:** разработка настольного приложения для криптографической защиты файлов (шифрование/расшифрование), генерации RSA-ключей и проверки целостности файлов через хеширование.

### Цель
Разработать программный продукт, обеспечивающий пользователю:
- конфиденциальность файлов при передаче/хранении;
- контроль целостности и снижение риска подмены файлов;
- удобство применения без специальных знаний.

### Задачи
1. Реализовать функции шифрования и расшифрования файлов.
2. Реализовать генерацию RSA ключей и работу с ними.
3. Реализовать вычисление хеш-функций для файлов.
4. Реализовать сравнение хешей двух файлов.
5. Реализовать хранение истории операций и экспорт в CSV.
6. Реализовать смену темы (светлая/тёмная).
7. Провести тестирование корректности.

### 4.1.1. Модель угроз

Проблема безопасности при работе с файлами во многом обусловлена спецификой современного цифрового обмена данными: высокая скорость передачи информации в сочетании с доверием к привычным каналам связи (электронная почта, мессенджеры, облачные хранилища) снижает бдительность пользователей. Пользователи склонны полагать, что файлы, полученные от знакомых источников или скачанные с проверенных сайтов, безопасны и не были изменены. Именно это доверие и используется злоумышленниками для компрометации данных или распространения вредоносного ПО.

**Объекты защиты**

В рамках данной модели мы выделяем четыре ключевых момента, требующих защиты:

1. **Конфиденциальность данных:** Файлы, содержащие персональную информацию, коммерческую тайну, финансовые данные, медицинские записи или другие чувствительные материалы, которые не должны быть доступны третьим лицам без авторизации.

2. **Целостность файлов:** Загруженные с интернета установщики, архивные файлы, документы и резервные копии могут быть подменены злоумышленниками на этапе передачи или хранения. Подмена файлов может привести к установке вредоносного ПО, потере данных или компрометации системы.

3. **Безопасность криптографических операций:** Пользователи часто выбирают слабые пароли для шифрования, теряют ключи шифрования или неправильно используют криптографические инструменты из-за недостаточной подготовки, что приводит к компрометации защищённых данных.

4. **Контроль и аудит операций:** Отсутствие истории выполненных криптографических операций создаёт риски невозможности восстановления доступа к файлам, потери данных о параметрах шифрования и невозможности проведения аудита безопасности.

**Суть угрозы:** Атакующий использует недостаточную бдительность пользователя и отсутствие доступных инструментов для защиты. Его цель — получить доступ к конфиденциальным данным путём перехвата незашифрованных файлов, подмены легитимных файлов на вредоносные, эксплуатации слабых паролей или использования ошибок пользователя при работе с криптографическими инструментами.

### 4.1.2. Соответствие тактикам и техникам MITRE ATT&CK

Ниже разобраны конкретные техники атак из базы MITRE ATT&CK, против которых направлена работа разрабатываемого приложения.

#### 1. T1040 - Network Sniffing

**Что соответствует:** Перехват незашифрованных файлов при передаче по незащищённым или скомпрометированным сетям. Злоумышленник, имеющий доступ к сетевому трафику (например, через публичный Wi-Fi, скомпрометированный маршрутизатор или перехват сетевых пакетов), может получить доступ к передаваемым файлам, если они не зашифрованы. Особенно актуально при передаче файлов через публичные Wi-Fi сети, учебные и корпоративные сети без end-to-end шифрования.


**Как противодействует проект:** Приложение предоставляет пользователю простой механизм шифрования файлов перед передачей через любые каналы связи. Файл, зашифрованный с использованием AES-256 или других стойких алгоритмов, становится недоступным для чтения злоумышленником даже при полном перехвате сетевого трафика. Пользователь может зашифровать файл перед отправкой и передать пароль получателю по защищённому каналу отдельно.

#### 2. T1565 - Data Manipulation

**Что соответствует:** Подмена файлов при скачивании из интернета или получении через ненадёжные каналы. Злоумышленник может заменить легитимный файл (например, установщик программы, архив с документами или резервную копию) на вредоносный или изменённый вариант, сохраняя при этом его имя и внешний вид.

**Как противодействует проект:** Приложение реализует функцию сравнения хеш-сумм (HashCompare), позволяющую пользователю проверить целостность скачанного файла. Если распространитель файла предоставил эталонную хеш-сумму (например, SHA-256), пользователь может вычислить хеш скачанного файла и сравнить его с эталоном. Несовпадение хешей однозначно указывает на подмену файла, и пользователь может отказаться от его использования, тем самым предотвращая компрометацию системы.

#### 3. T1552.001 - Unsecured Credentials: Credentials In Files / Weak Passwords

**Что соответствует:** Использование слабых паролей для защиты конфиденциальных файлов. Пользователи часто выбирают простые, предсказуемые пароли (например, "12345", "password", даты рождения), которые легко поддаются перебору (brute-force) или словарным атакам. Это делает шифрование практически бесполезным, так как злоумышленник может восстановить пароль за короткое время.

**Как противодействует проект:** Приложение включает механизм валидации стойкости пароля (`PasswordStrength.Evaluate`), который оценивает сложность пароля по нескольким критериям (длина, разнообразие символов, наличие цифр, специальных символов). Слабые пароли (score ≤ 1) блокируются для использования в операциях шифрования, если не включён флажок "Allow weak password". Визуальный индикатор стойкости пароля (`PwdStrengthBar`) в реальном времени показывает пользователю уровень защиты выбранного пароля, побуждая выбирать более стойкие варианты.

#### 4. T1566.001 - Phishing: Spearphishing Attachment

**Что соответствует:** Распространение вредоносных файлов под видом легитимных документов. Злоумышленник маскирует исполняемые файлы или вредоносные скрипты под офисные документы (PDF, Word, Excel) или архивы, используя двойные расширения (например, `документ.pdf.exe`) или поддельные иконки, чтобы обмануть пользователя и заставить его запустить вредоносный код.

**Как противодействует проект:** Хотя приложение не выполняет антивирусную проверку файлов напрямую, оно предоставляет инструмент для проверки целостности файлов через хеширование. Если пользователь получает файл от ненадёжного источника и сомневается в его подлинности, он может запросить у отправителя хеш-сумму файла и проверить её с помощью функции HashCompare. Это позволяет выявить подмену файла даже в случаях, когда внешний вид файла соответствует ожидаемому.

### 4.1.3. Вывод по модели угроз

Разрабатываемое приложение работает как инструмент превентивной защиты (направление Blue Team), направленный на повышение уровня практической защищённости конечного пользователя. Оно закрывает уязвимости, связанные с человеческим фактором, недостаточной криптографической грамотностью и отсутствием доступных инструментов для защиты данных.

**Целевой результат:** Ключевая задача — предотвратить возможную компрометацию данных на этапе хранения и передачи, обеспечив пользователю простые и надёжные инструменты для шифрования файлов и проверки их целостности. Предоставляя доступные криптографические функции с минимизацией ошибок и полной прозрачностью операций, приложение значительно снижает риск успешной реализации атак, направленных на нарушение конфиденциальности и целостности пользовательских данных.

## 4.2. Описание алгоритмов и логики работы приложения

Приложение — WPF-клиент, который выполняет криптооперации через запуск `openssl.exe` как внешнего процесса. Все действия фиксируются в журнале `history.jsonl` (формат JSONL).

### 4.2.1. Архитектура обработки команд
Приложение использует унифицированный подход для выполнения криптографических операций:
- Формируется строка аргументов `args` для `openssl.exe`.
- Метод `RunOpenSSLAsync(args)` запускает процесс через `ProcessStartInfo` с перенаправлением потоков вывода.
- Синхронно собирается результат и выводится в `OutputBox`.
- Результат логируется в историю операций.

Режимы работы (переключатель `ModeList`):
- **Encryption** — шифрование/расшифрование файлов
- **Hashing** — хеширование и сравнение хешей
- **Keys** — генерация RSA-ключей
- **History** — управление историей операций

### 4.2.2. Валидация пароля
Перед операциями шифрования вызывается `EnsurePasswordOk()`:
- Проверяется, что пароль не пустой.
- Оценивается сложность через `PasswordStrength.Evaluate()`.
- Если пароль слабый (score ≤ 1), операция блокируется, если не включён флажок `Allow weak password`.
- Индикатор `PwdStrengthBar` обновляется в реальном времени.

### 4.2.3. Криптографические операции
**Шифрование (Encrypt):**  
Формируется команда `openssl {algo} [-salt] -in "файл" -out "файл.enc" -k "пароль"` и выполняется асинхронно.

**Расшифрование (Decrypt):**  
Аналогично шифрованию с флагом `-d`, результат сохраняется как `файл.dec`.

**Хеширование (Hash):**  
Команда `openssl dgst -{algo} "файл"` вычисляет хеш. Время выполнения замеряется через `Stopwatch`.

**Сравнение хешей (HashCompare):**  
Запускаются две команды хеширования параллельно. Хеши извлекаются функцией `ExtractDigestFromOpenSslOutput()` и сравниваются без учёта регистра. Результат выводится как "MATCH" или "DIFFERENT".

**Генерация RSA-ключей (KeyGen):**  
Команда `openssl genrsa -out "файл.pem" {bits}` генерирует закрытый ключ размером 2048/3072/4096 бит.

### 4.2.4. Управление историей и интерфейсом
История операций сохраняется в `history.jsonl`:
- При старте приложения вызывается `logger.Load()` и привязка к `HistoryGrid`.
- При каждой операции вызывается `logger.Append(LogEntry)`.
- Пользователь может очистить историю или экспортировать её в CSV через `logger.ExportCsv(path)`.

**Переключение темы:** Кнопка `ToggleTheme_Click` изменяет `_isDarkTheme` и перезагружает XAML-ресурсы (`Dark.xaml` или `Light.xaml`).

## 4.3. Описание существующих алгоритмов
В приложении используются готовые реализации OpenSSL:
- **Симметрическое шифрование** (выбор в `AlgoBox`) для защиты конфиденциальности файлов при передаче/хранении;
- **RSA** (генерация пары ключей `genrsa`) для задач, где требуется асимметрическая криптография;
- **Хеш-функции** (`openssl dgst -<algo>`) для контроля целостности и обнаружения подмены.

Важно: хеширование не скрывает содержимое файла, а позволяет выявлять изменения.

## 4.4. Нормативно-правовые аспекты и безопасная эксплуатация
Приложение повышает защищённость пользовательских данных за счёт:
- шифрования конфиденциальных файлов (снижение риска утечки);
- контроля целостности скачанных файлов через хеширование (снижение риска подмены).

При работе с файлами, содержащими персональные данные, использование шифрования и контроль целостности соответствует общим принципам безопасной обработки данных (в т.ч. в контексте 152-ФЗ).

# 7. Дизайн продукта творческого проекта

## 7.1. Новизна и оригинальность

### Научная новизна
Проект представляет новый подход к проблеме повышения криптографической грамотности массового пользователя. В отличие от существующих решений, которые либо ориентированы на специалистов (CLI-инструменты), либо перегружены функциями (как Kleopatra), приложение реализует концепцию **адаптивной криптографии** — приведение сложных криптографических операций в соответствие с когнитивными способностями и практическими потребностями неподготовленного пользователя.

### Практическая новизна
1. **Интегрированный интерфейс:** реализована интеграция пяти ключевых криптографических операций (шифрование, расшифрование, хеширование, сравнение хешей, генерация ключей) в едином, последовательно навигируемом интерфейсе без перегрузки функциями.

2. **Валидация стойкости пароля в реальном времени:** механизм `PasswordStrength` с визуальным индикатором предотвращает использование слабых паролей до выполнения операции, что значительно снижает риск компрометации.

3. **Система логирования операций с экспортом:** формат JSONL и экспорт в CSV позволяют конечным пользователям и организациям вести аудит криптографических операций — функция, отсутствующая в Kleopatra и онлайн-инструментах.

4. **Прозрачность команд:** функция `Show Command` выводит точные команды OpenSSL, что повышает доверие пользователя к безопасности и позволяет изучить криптографию на практике.

- Все вместе в одном простом приложении в удобном виде это реализовано впервые

### Инновационность подхода
- **Снижение барьера входа:** приложение требует минимальной криптографической подготовки, в то время как CLI-инструменты требуют знаний параметров и синтаксиса.
- **Минимизация человеческих ошибок:** валидация путей файлов, проверка существования, предупреждения об ошибках, подсказки в интерфейсе.

### Вклад в информационную безопасность
Проект вносит вклад в **прикладную ИБ** путём:
- повышения практического уровня защиты данных в организациях и школах;
- создания учебного инструмента для демонстрации принципов криптографии;
- подтверждения гипотезы, что хорошо спроектированный интерфейс может значительно повысить безопасность на уровне пользователя.

- Проверка механизма валидации пароля (PasswordStrength.Evaluate)

- Тестирование логирования (OperationLogger)

**Интеграционное тестирование:**
- Проверка взаимодействия MainWindow ↔ OpenSSL.exe
- Проверка асинхронности (async/await в RunOpenSSLAsync)
- Проверка потокобезопасности UI

**Регрессионное тестирование:**
- Сравнение результатов с эталонными значениями из OpenSSL CLI v1.1.1
- Повторное тестирование после каждой доработки

### 8.1.2 Тестовые сценарии и данные

| Операция | Тестовые данные | Ожидаемый результат |
|----------|-----------------|--------------------|
| Encrypt | Файлы 1 КБ, 10 МБ, 100 МБ (текст, бинарь) | .enc файл, не читаемый без пароля |
| Decrypt | .enc файлы с разными алгоритмами | Исходный файл восстановлен полностью |
| Hash | Файлы SHA256, SHA384, SHA512, MD5 | Хеш совпадает с OpenSSL CLI |
| HashCompare | Идентичные файлы, изменённые (1 байт) | MATCH / DIFFERENT |
| KeyGen | RSA 2048, 3072, 4096 бит | .pem файл, валидный ключ |

### 8.1.3 Инструменты тестирования

- **OpenSSL CLI v1.1.1** — эталон для сравнения результатов
- **Stopwatch** — замер времени выполнения операций
- **Логирование в OutputBox** — отслеживание выполнения операций
- **Ручное тестирование** — проверка UI-элементов и асинхронности

## 8.2. Результаты и анализ тестирования

### 8.2.1 Тестирование криптографических операций

**Шифрование/расшифрование (Encrypt/Decrypt):**
- Протестировано 5 алгоритмов: AES-128, AES-192, AES-256, ChaCha20, 3DES
- Файлы 1 КБ, 10 МБ, 100 МБ шифруются и расшифровываются корректно
- С флагом `-salt` добавляется соль, что повышает безопасность против словарных атак
- Результаты соответствуют эталонам OpenSSL CLI (100% совпадение)

**Хеширование (Hash):**
- Протестировано 4 алгоритма: SHA-256, SHA-384, SHA-512, MD5
- Хеши совпадают с `openssl dgst` на 100%
- Изменение 1 бита файла приводит к полной смене хеша
- Замер времени фиксируется корректно

**Сравнение хешей (HashCompare):**
- Идентичные файлы показывают совпаденние
- Изменённые файлы (даже на 1 байт) показывают различие
- Два независимых запуска OpenSSL дают согласованные результаты
- Результаты логируются в историю

**Генерация RSA-ключей (KeyGen):**
- Ключи 2048, 3072, 4096 бит генерируются корректно
- Ключи сохраняются в формате PEM
- Ключи валидны для использования в OpenSSL

### 8.2.2 Тестирование валидации и обработки ошибок

- Пустой пароль блокирует операцию с предупреждением
- Слабый пароль (score ≤ 1) блокирует операцию, если не включён флажок
- Несуществующие файлы перехватываются до запуска OpenSSL
- Неправильные пути отклоняются с сообщением об ошибке
- Ошибки OpenSSL выводятся в OutputBox
- Приложение не крашится при любых ошибках

### 8.2.3 Тестирование асинхронности и UI

- Большие файлы (100 МБ+) обрабатываются асинхронно, UI не блокируется
- Переключение режимов работает плавно
- Смена темы (Light/Dark) отображается мгновенно
- OutputBox прокручивается автоматически
- StatusText обновляется в реальном времени

### 8.2.4 Тестирование логирования и экспорта

- Все операции логируются в history.jsonl
- История сохраняется между перезагрузками приложения
- Экспорт в CSV создаёт читаемый файл с колонками: Operation, InputPath, OutputPath, Algorithm, Status, Message, Timestamp
- CSV открывается корректно в Excel и текстовых редакторах
- Функция очистки истории работает с подтверждением

### 8.2.5 Тестирование граничных случаев

| Сценарий | Результат | Статус |
|----------|-----------|--------|
| Файл > 1 ГБ | Асинхронная обработка, UI не блокируется | Пройдено |
| Пароль со спец. символами | Корректно передаётся в OpenSSL | Пройдено |
| Путь с кириллицей | Файлы обрабатываются корректно | Пройдено |
| OpenSSL отсутствует | error "openssl.exe not found" | Пройдено |

**Итого:** 25+ тест-кейсов, 100% успешны, 0 критических дефектов.

4. **Нет поддержки цифровой подписи:** приложение проверяет целостность, но не подлинность. Решение: добавить в версию 2.0.
5. **UI только для Windows:** привязана к WPF. Решение: портировать на .NET MAUI для кроссплатформенности.
